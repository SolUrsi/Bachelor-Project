   ?? Before: Manual reconnection attempts in game code
   ?? After:  Automatic reconnection with exponential backoff

INTERFACE-BASED DESIGN
???????????????????????????????????????????????????????????????????????????????

EventService depends on TWO interfaces:

  1. ISessionManager
     ?? GetSessionId()
     ?? GetScenarioId()
     ?? GetScenarioName()
     
     Implemented by: TrainingSessionManager
     
  2. IMqttEventPublisher
     ?? PublishEvent(TrainingEvent evt)
     ?? PublishScoreRequest(ScoreRequestMessage request)
     
     Implemented by: MqttEventPublisher

This allows:
  • Easy testing (mock implementations)
  • Easy extension (alternative implementations)
  • Loose coupling (EventService doesn't know concrete types)
  • Dependency injection without DI container

WHAT EACH COMPONENT DOES
???????????????????????????????????????????????????????????????????????????????

MqttApplicationManager
  Role: Application-wide MQTT transport
  Responsibility:
    • Create and maintain broker connection
    • Handle automatic reconnect
    • Queue outgoing messages (thread-safe)
    • Receive incoming messages
    • Fire events for listeners
  Lifespan: Application lifetime (singleton, DontDestroyOnLoad)

MqttEventPublisher
  Role: Serialization layer
  Responsibility:
    • Convert TrainingEvent to JSON
    • Convert ScoreRequestMessage to JSON
    • Route to MqttApplicationManager
  Dependency: MqttApplicationManager.Instance
  Lifespan: Application lifetime

EventService
  Role: Centralized public API for game systems
  Responsibility:
    • Build TrainingEvent objects
    • Provide clean methods (PublishHazardMarked, etc.)
    • Resolve SessionManager for context
    • Resolve MqttEventPublisher for sending
  Dependencies: ISessionManager, IMqttEventPublisher
  Lifespan: Application lifetime (finds implementations at Start)

ScoreResponseHandler
  Role: Backend score listener
  Responsibility:
    • Subscribe to MqttApplicationManager messages
    • Parse score responses
    • Fire OnScoreReceived event
  Dependency: MqttApplicationManager.Instance
  Lifespan: Application lifetime

TrainingSessionManager
  Role: Session context provider
  Responsibility:
    • Implement ISessionManager
    • Create and manage SessionContext
    • Track session ID, scenario ID, scenario name
  Dependency: None
  Lifespan: Application lifetime (can be recreated per game)

SessionContext
  Role: Plain data holder
  Responsibility:
    • Hold session ID, scenario info
    • Provide getter methods
    • Reset for new sessions
  Dependency: None (pure data class)
  Lifespan: One per session

SUMMARY
???????????????????????????????????????????????????????????????????????????????

The refactored architecture provides:

? Persistent MQTT connection (always active)
? Clean separation of concerns (Core vs. Training)
? Centralized event publishing API (EventService)
? Automatic reconnection (robust)
? Thread-safe async operations (performant)
? Interface-based design (testable, extensible)
? No scene-load dependencies (scalable)
? Easy to debug (clear log messages)

The system is now production-ready and can scale to any number of scenes,
interactable objects, and event types without modification.
